# 股市复盘

根据复盘对象日期的盘面表现、盘中及盘后消息，生成标准化的复盘报告。支持两种运行模式。

> **Cron 部署层细节**（互斥锁、final response 推送语义、看门狗重试、节假日时间框架、推送/上报语义等）已提取至 [`references/cron_setup.md`](./references/cron_setup.md)。本文件保留 Skill 本体职责，不再堆叠部署层内容。

---

## 1. 🔴 核心约束（不可违反）

以下规则优先级最高，任何情况下不得违反：

### 1.1 事实约束
- **所有写入报告的数据必须来自本次实际采集的结果。** 严禁复用示例、模板、历史复盘或对话记忆中的任何数据。
- **无法核实的内容必须省略或标注为「未确认」。** 严禁为了完整性编造板块涨跌幅、个股涨跌幅、新闻标题或任何看似合理的数据。
- **模板、示例、历史输出仅用于说明结构，不是事实来源。** 如果找不到对应事实，宁可留空也不编造。

#### 1.1.1 🔴 反"截取式漏报"规则（2026-06-29 核聚变漏报事件新增）

**事件回顾**：6/26 财联社原文「国家发改委、国家能源局印发《新型能源体系建设「十五五」规划》，**强化可控核聚变**、太空电站、高温超导输电等理论研究和技术创新」——本应至少 3 个产业方向（核聚变/超导/太空电站）单列，但 6/26 复盘和 6/29 早盘都只提取了"电网投资 5 万亿+"一个角度，**核聚变/超导/太空电站被当作次要内容丢弃**。结果 6/29 盘中核聚变板块（百利电气/联创光电/中核科技/中国核建）涨停，主人错过盘前最佳介入点。

**根因（避免误诊）**：这不是 LLM "凭印象编"问题——核聚变是真实存在的政策原文。**真正的根因是消息面总结阶段选择性截取**：政策原文含并列方向时，LLM 倾向于只挑数字最大/最显眼的那个，其余当作"次要"丢弃。这与反编造红线**正相反**——反编造是禁止编造数据，这是禁止**遗漏数据**。

**禁止行为**：
- ❌ **政策类消息原文含多个产业方向时，只挑一个写、其余丢弃**（如"电网投资 5 万亿+" vs "核聚变+超导+太空电站"）
- ❌ **用"等"字合并多个产业方向**（"电网投资 5 万亿+ 等"——会让人误以为"等"涵盖了什么，实际什么也没涵盖）
- ❌ **政策原文出现"强化 XX、YY、ZZ"或"加快 AA、BB、CC"等并列结构时，只提第一个**

**强制规则**：
- ✅ **政策/规划/会议消息原文若列举 ≥3 个并列方向，每个方向必须单列一条**进入消息面清单
- ✅ **每条必须包含：原文摘要（≤30 字）+ 受益板块/概念 + 至少 1 个代表个股（如有）**
- ✅ **复盘文件中"消息面"章节如发现多方向被合并为单条，必须拆开重写**
- ✅ **早盘快报/复盘必须在「今日关注板块」中标记所有"政策点名但未发酵"的方向**（即使当日无板块异动，也要作为潜在催化方向列出）

**反例（❌ 错误摘录，2026-06-26/06-29 实测）**：
```
- 国家发改委、国家能源局印发新型能源体系建设「十五五」规划，电网投资 5 万亿+
```
（实际原文含 4 个方向：电网投资 5 万亿+/可控核聚变/太空电站/高温超导输电——只提了 1 个）

**正例（✅ 正确摘录）**：
```
- 国家发改委、国家能源局印发新型能源体系建设「十五五」规划：
  • 电网投资 5 万亿+（国家能源局表示十五五期间全国电网固定资产投资将达 5 万亿元以上）
  • 强化可控核聚变理论研究和技术创新（概念股：百利电气/联创光电/中核科技/中国核建）
  • 太空电站、高温超导输电纳入规划
- 第三批超长期特别国债支持消费品以旧换新资金已下达
```

**自检清单**（消息面章节生成后必走）：
```
□ 政策原文是否含并列结构（"AA、BB、CC"或"加快 AA、BB、CC"）？
  → 是：每个并列项必须单独列一条
□ 是否有"等"字合并多个方向？
  → 是：拆开重写，禁止"等"合并
□ 摘录中是否只保留了第一个方向而丢失其他？
  → 是：补全所有方向
□ 「今日关注板块」是否覆盖了所有"政策点名但未发酵"的方向？
  → 缺失：补充并标注"政策点名·待发酵"
```

**相关教训：与 cron final response 一致性配合使用**。如果 markdown 文件漏写了某些政策方向，**不要**让 cron final response 阶段"凭对话历史印象"补回来——**回到 markdown 重新写**。详见 [`references/cron_setup.md`](./references/cron_setup.md) 「final response 推送语义」章节。

### 1.2 🔴 反编造红线（2026-06-26 涛涛车业事件新增）

**绝对禁止**以下行为（违反任何一条都视为严重事故）：

#### 1.2.1 A. 量化数据的编造
- ❌ **禁止编造「XX%」「XX倍」「历史高位/低位」类量化结论**而不提供数据来源。哪怕数字看似可以从其他数据推导（如「现价/发行价-1」），只要该数字未在本次分析中实际拉取历史数据验证，就**视为编造**。
- ❌ **禁止用「时间跨度过大的数据假装是近期数据」**。例如：把 IPO 至今涨幅当作 60 日涨幅、把 5 年平均 PE 当作当前 PE 分位。
- ❌ **禁止「拍脑袋式风险评估」**：未实际拉取数据就写「风险信号触发」「股价已超涨」「估值偏高」等定性结论。
- ❌ **禁止「为了凑齐清单而虚构素材」**：风险提示有明确的触发清单（如 R3-1 要求 60日涨幅≥40%），没有真实数据证明触发条件时，**必须沉默**，而不是伪造触发依据。

#### 1.2.2 B. 数据来源的真实性
- ❌ **禁止「数字正确 ≠ 事实正确」的伪装**。例如：发行价73.45、现价227.83 是事实，但「60日涨幅210%」是编造——**计算过程合法但输入数据/口径错误，依然是编造**。
- ❌ **禁止「看似有出处但实际无关」的论证**。例如：用 IPO 定价推导近期涨幅、用全年数据推算单季表现。
- ❌ **禁止「模糊化处理掩盖数据缺失」**：写「近期涨幅较大」「估值偏高」「市场情绪谨慎」等模糊措辞来掩盖「没有真实数据」。**模糊措辞本身就是编造的变种**。

#### 1.2.3 C. 输出前的强制自检清单（每次输出前必走）

每次输出涉及任何量化结论（涨幅、跌幅、估值倍数、风险等级判定）前，必须在内心完成以下自检：

```
□ 这条数据的原始来源是什么？API返回值？浏览器抓取页面哪一行？公告第几段？
□ 我能在本次会话中找到该数据的实际来源吗？
□ 如果找不到，我应该写「未核实」「未获取」「无法确认」吗？
□ 这条结论是用「相关但不同口径」的数据假装出来的吗？
```

**4 个勾中任一为否 → 不得输出该结论。**

**强制自检脚本**（推荐每次输出前调用）：
```bash
python scripts/anti_fabrication.py checklist   # 朗读清单 + 模糊措辞黑名单
python scripts/anti_fabrication.py check-claims claims.json   # 校验量化声明必须有 source/scope/timestamp
```

#### 1.2.4 D. 「逻辑推演」与「事实陈述」的强制区分

输出时必须明确标注两类内容：

| 类型 | 标识 | 示例 |
|------|------|------|
| **事实陈述** | 直接陈述，标注数据来源 | 「2026 Q1 营收 10.59 亿（新浪财经财务摘要）」|
| **逻辑推演** | 标注「【假设】」「【推测】」前缀 | 「【假设】关税前置囤货可能推高 Q1 营收」|

**禁止把「逻辑推演」包装成「事实陈述」**，尤其禁止用「数据上看」「客观分析」「市场认为」等措辞包装主观推断。

### 1.3 标题约束
- **标题格式不可变。** 当日复盘强制 `YYYY年M月D日（周X）A股复盘`，早盘快报强制 `YYYY年M月D日早盘快报`。不得添加副标题、使用短日期格式或调换词语顺序。

### 1.4 JSON 类型约束
- **`markets` 必须是对象**（含 `summary`、`indices`、`volume` 三个字段），不是数组。
- **`todayHot` 必须是对象**（含 `topSectors`、`concepts`、`fallingSectors`、`summary` 四个字段），不是 `null`。早盘快报无盘中数据时也必须设为含空数组的对象。
- **所有 `changePercent` 字段必须是数字（number），不是字符串 `"N/A"` 或 `null`。** 无数据时填 `0` 并标注「基于代表性个股均值估算」。
- **`content` 字段必须与 markdown 正文一致，不得留空。**
- **JSON 必须用 `python3` 的 `json.dump(data, f, ensure_ascii=False, indent=2)` 生成，不得手写。** 避免中文引号导致解析失败。

### 1.5 数据源约束
- **以下渠道已确认不可用，严禁调用**：akshare 东财行业/概念板块（走的同一后端被封）、akshare THS 同花顺板块排名（同花顺改页面结构失效）。**⚠️ 东财 push2 API 状态不稳定**——2026-06-25 / 06-30 短时间内可用，**2026-07-01 起完全不可用**（5+ 次重试均 `Remote end closed`/Empty reply，疑似 IP 段屏蔽）。应将其视为「可尝试」而非「可靠」来源，失败时立即回退至**Sina `/cpbd/` 操盘必读全文（15:15 收盘后稳定）+ Sina 首页标题 + Sina snipe 实时异动文章 + 腾讯 API pt 板块代码**组合获取板块数据。详细回退矩阵见下方「东财 push2 API 当前状态」章节。
- **板块涨幅≠个股涨幅。** 个股涨跌幅必须来自腾讯行情 API 或 pyTDX 的实际拉取，严禁用板块涨幅推算个股涨幅。
- **腾讯 API 的成交额字段（索引7）不可靠**——主力指数均返回 `"0"`。使用成交量字段（指数索引6，单位「手」/100 股）描述市场活跃度，严禁编造成交额数字。

### 1.6 输出章节约束
- **当日收盘复盘不包含「明日关注板块」和「明日关注个股」章节。** 这两个章节仅出现在早盘快报中。
- **早盘快报必须包含「今日关注板块」和「今日关注个股」章节。**

### 1.7 查询约束
- **股价查询必须输出三条核心数据：现价、涨跌幅（%）、涨跌额（绝对值）。** 三列缺一不可，不可用「—」或「N/A」占位。境外股票用 Yahoo Finance API，A 股用腾讯行情 API。

### 1.8 上报约束
- **token 预检必须在完整复盘生成之前完成。** 先用最小 payload（`{"date":"...","content":"ping"}`）POST 测试，确认 code 400（鉴权通过）后再生成完整报告。code 401 立即告知用户 token 过期。
- **JSON 中必须包含 `type` 字段**：早盘快报填 `1`，当日复盘填 `2`。2026-06-29 实测 API 要求必填——缺失返回 code 400 "type 字段必须为 0（自动）、1（早盘快报）或 2（今日复盘）"。在 Python 脚本生成 JSON 时添加 `data["type"] = 1`（或 `2`）再 `json.dump`。
- **API 上报方式（按优先级）**：
    - **首选：Python 脚本上报（cron 模式下推荐）** — `write_file` 写入 Python 脚本到 `/tmp/upload.py`，用 `urllib.request` 发送 POST，API key 从 `~/.profile` 逐行读取。无 shell 变量替换问题，无 `Bad substitution` 风险。2026-06-26 早盘快报实测验证通过。
    - **备选：terminal curl 上报** — `source ~/.profile && curl -X POST ... -H "Authorization: Bearer $STOCK_REVIEW_API_KEY" -d @review.json`。注意 `xntk_` 前缀的 API key 可能因 shell 特殊字符触发 `Bad substitution` 错误，此时应改用 Python 脚本方案。

### 1.9 🔴 风险信号触发机制（不可绕过）

**触发源**：早盘快报、个股推荐、持仓诊断、板块推荐等任何涉及**「今天/明日该买什么」**的输出。

**核心原则**：**重大风险信号必须先于推荐结论触发，强制覆盖乐观叙事**。即使长期逻辑成立，短期风险等级一旦达到阈值，必须在输出最前面给出风险提示，并降低推荐仓位倾向（默认不推荐买入，改为「观望」或「减仓」）。

#### 1.9.1 风险等级定义（严格按数值，不允许主观降级）

**编号规则**：**数字越↑越严重**（R1 最严重，R4 是常态）。

| 等级 | 标签 | 触发条件（满足任一即触发） | 默认操作建议 |
|------|------|---------------------------|--------------|
| **🔴 红色·极高风险** | **R1** | 见下方 R1 触发清单（战争/熔断/加息突变/政策超预期） | **严禁推荐买入；只提示减仓/清仓/观望** |
| **🟠 橙色·高风险** | **R2** | 见下方 R2 触发清单（夜盘跌≥1.5%/主力连撤/政策博弈期临近落地） | **谨慎推荐；必须并列展示风险提示与机会** |
| **🟡 黄色·中等风险** | **R3** | 单一不利信号（如个股前期涨幅过大、单一板块龙头跌停、汇率单日异动≥1%） | **正常推荐；附一句风险提示即可** |
| **🟢 绿色·常态** | **R4** | 无任何风险信号触发 | **正常推荐** |

#### 1.9.2 R1（红色·极高风险）触发清单——满足任一即触发，不可协商

**A. 地缘/军事冲突类**：
- 中东/欧洲/台海/南海**爆发实际军事冲突**（非「局势紧张」式表述，需有实际交火/导弹/制裁生效等事实）
- 联合国安理会通过**涉中俄重大制裁决议**
- 任一**核武国家**进入战时状态或进行核试验

**B. 全球金融市场熔断/异动类**：
- 全球任一主要市场（美/欧/日/韩/港/台）**单日触发熔断**（包括盘前熔断、盘中熔断、夜盘熔断）
- VIX 恐慌指数**单日涨幅 ≥30%**（从正常区间 15-20 跳到 22+）
- 美股**三大指数跌幅 ≥3%**（科技股 ≥5%）
- 美元指数**单日涨幅 ≥1.5%**（压制所有大宗商品）
- 原油**单日涨幅 ≥8%**（引发通胀恐慌）

**C. 美联储/全球央行异动类**：
- 美联储官员**公开反转立场**（如鹰转鸽/鸽转鹰，且为非例行会议期间的突发表态）
- 美联储**紧急降息或加息**（非议息会议日）
- 中国央行**意外调整 MLF/逆回购利率 ±10bp 以上**

**D. 重大政策超预期类**：
- 美国对华**新增重大关税/制裁/出口管制**（232/301/SDN 名单新增）
- 中美**金融战/科技战升级**（如美对华投资禁令、SWIFT 制裁传闻）
- 国内**突发重大监管**（如教培/平台经济/医药集采突发反转）

**E. 系统性风险类**：
- 国内任一**大型金融机构暴雷**（银行/券商/保险/信托）
- **汇率单日异动 ≥1.5%**（离岸人民币对美元）
- **主权信用评级**被三大评级机构下调

#### 1.9.3 R2（橙色·高风险）触发清单——满足任一即触发

- **商品夜盘集体大跌**：沪铜/沪铝/沪金/沪银/螺纹钢任一主力合约夜盘跌幅 ≥1.5%（含夜盘累计）
- **主力资金连续2日净流出某板块**（净流出额 ≥30亿元/日）
- **政策博弈期临近落地**（如重大会议、关税审查、议息会议 ≤3 个交易日）
- **关键数据公布前 24 小时**：CPI/PPI/PMI/非农就业数据公布前夜
- **期权市场 Put/Call 比率突变**（如从正常 0.7-1.0 跳到 ≥1.5）
- **国内期指 IC/IM 贴水扩大 ≥1%**（小盘股风险偏好下降）
- **北向资金单日净流出 ≥80亿元**（外资撤离信号）
- **行业龙头股跌停**（板块内市值 TOP3 任一跌停）

#### 1.9.4 R3（黄色·中等风险）触发清单

- **个股前期涨幅过大**：60 日涨幅 ≥40% 且无业绩支撑
- **单一板块龙头跌停**（非 TOP3 市值）
- **汇率单日异动 0.5%-1%**
- **商品夜盘单品种跌 0.5%-1.5%**
- **VIX 单日涨 15%-30%**

#### 1.9.5 风险提示输出格式（强制模板）

**触发后必须按以下格式插入到输出最前面**，不允许弱化语言（禁止用「可能」「或许」「值得关注」等模糊措辞）：

```
🚨 【风险提示·{R1/R2/R3}·{日期}】

{具体风险事件，列事实，不列预测}

📌 影响：{直接说明对当前讨论板块/个股的具体影响}

⚠️ 操作建议：{严格按对应等级的执行清单，不可偏离}
```

**R1 操作建议模板**（必须选其一，不允许"两边都对"的表述）：
- 「今日**严禁建仓**，持有者建议**减仓 50%+ 或清仓观望**，等风险事件消化后再评估」
- 「今日**严禁推荐买入**，关注盘后/隔夜事件演变，明日开盘前再决策」

**R2 操作建议模板**：
- 「今日可谨慎参与，但**仓位上限 50%**，且必须设止损线（建议 -3%）
- 「板块博弈期，**建议持有者减仓 1/3**，未持有者**继续观望**」

**R3 操作建议模板**：
- 「正常参与，但注意**单一标的仓位 ≤ 20%**，分散风险」

#### 1.9.6 风险信号采集路径（必须实际验证，不可省略）

**每次涉及推荐前，按以下顺序扫描风险信号**（每条都需有来源）：

- **隔夜美股**（道指/纳指/标普/VIX/美元指数/原油）— 数据源：财联社电报 + 新浪操盘必读文章
- **商品夜盘**（沪铜/沪铝/螺纹钢/铁矿/黄金）— 数据源：财联社电报关键词「夜盘」
- **全球风险事件**（战争/制裁/熔断）— 数据源：财联社电报 + 新浪 7x24
- **国内政策事件**（重大会议/政策/数据公布）— 数据源：新浪财经首页标题
- **主力资金动向**（板块/个股净流入流出）— 数据源：财联社主力资金监控电报
- **北向资金** — 数据源：财联社电报 + 东方财富

**采集到的信号必须能在原始链接/标题/电报中找到对应事实**，不得主观推断。

#### 1.9.7 反模式（明确禁止的输出行为）

- ❌ 「长期看好基本面，短期波动可忽略」 — 当 R1 触发时绝对禁止
- ❌ 「技术上已经超跌，有反弹需求」 — 用技术面掩盖基本面利空
- ❌ 「政策博弈期正是布局良机」 — R2 触发时绝对禁止
- ❌ 「市场恐慌正是机会」 — 主观判断恐慌 vs 实际风险，禁用
- ❌ 风险提示放在输出末尾或用小字 — 必须放最前面且醒目
- ❌ 风险提示用疑问句或假设句（「如果...可能...」「或许...值得关注」）— 必须用陈述句列事实

#### 1.9.8 阈值原则（防止警报疲劳）

**重大风险提示仅在风险等级达到阈值时触发**，不允许「日常絮叨式风险提示」：

- ✅ **只在 R1/R2/R3 触发时**输出完整 🚨 风险提示块
- ❌ **常态 (R4)** 下不要每段都加「注意风险」「仅供参考」等口头禅式风险提示
- ❌ **不允许「轻度信号也输出 R1 模板」** —— 信号达不到阈值就保持沉默，不要为了「显得严谨」而触发
- ❌ **不允许「同一信号反复触发」** —— 同一 R2 信号在 24 小时内最多触发 3 次，避免刷屏

**理由**：过度触发会使用户对风险提示麻木，等真正风险来临时反而被忽略。**精准触发**比**频繁触发**更重要。

**反例**：每次推荐都加「以上仅供参考，投资有风险，入市需谨慎」——这种提示属于噪音，违反阈值原则，应当删除。

## 2. 模式

> **章节分工**：本节仅定义各模式的「复盘对象 + 内容重点 + 标题格式 + 时间敏感陷阱」。输入约定（消息收集周期）见 [4]；输出字段要求见 [5]；执行步骤见 [6]。标题格式总约束见 [1.3]。

### 2.1 当日复盘（默认模式，15:15 执行）
- 复盘对象：当日盘中交易数据（9:30–15:00）
- 内容重点：指数表现、板块涨跌、热点个股、龙头梳理、盘中消息
- 不包含「明日关注板块/个股」——见 [1.6]
- 标题：`YYYY年M月D日（周X）A股复盘`（总约束见 [1.3]）
- 输入约定见 [4.2]；输出要求见 [5.2]；执行步骤见 [6.2]

### 2.2 早盘快报（8:00 执行）
- 复盘对象：前日收盘后至当前时刻（当日 8:00）的消息面
- 内容重点：隔夜美股、盘前消息、今日关注板块/个股
- 不包含当日盘中数据；关注结论必须综合「昨日盘面 + 隔夜消息」双依据（见 [4.3]）
- 标题：`YYYY年M月D日早盘快报`（总约束见 [1.3]）
- **执行时间敏感陷阱**：若早盘快报在实际当日收盘后（15:00+）执行，`qt.gtimg.cn` 等行情 API 查询个股会返回**当日收盘价**而非前一日数据，不可混用。必须优先从昨日复盘 JSON 读取历史收盘价；若只能靠 API，则必须在查询时标注日期。详见「早盘快报时间错位」陷阱。
- 输入约定见 [4.3]；输出要求见 [5.3]；执行步骤见 [6.3]

### 2.3 快速股价查询
- 触发：用户直接报出个股名称或代码（如「查一下茅台」「光迅科技 工业富联 行情」「宁德时代多少钱」「三星今天涨幅」）
- 覆盖范围：**A 股、港股、美股、日股等全球主要市场**
- 三条核心数据：① 现价 ② 涨跌幅（百分比）③ 涨跌额（绝对值）——约束见 [1.7]
- 输出：简洁内联表格，不生成本地文件、不上报 API
- 数据源：
  - A 股：腾讯行情 API `qt.gtimg.cn` + 东财公告 API `np-anotice-stock`
  - 境外（港股/美股/日股等）：
    - 首选：**yfinance**（`pip3 install yfinance`）。用法 `ticker.history(period='2d')` 获取日线，或 `ticker.info` 获取实时行情。⚠️ Yahoo 有频率限制，短时间内大量请求会触发 `YFRateLimitError`，需等待冷却。
    - 回退：Yahoo Finance v8 API `query1.finance.yahoo.com/v8/finance/chart/{symbol}?interval=1d&range=2d`。
    - 代码格式：港股 `0700.HK`、美股 `AAPL`、日股 `7203.T`。
  - 境外股票公告不可用，省略公告列
- 单次 `execute_code` 完成全部采集+输出，不需要浏览器
- 执行步骤见 [6.4]

## 3. 何时使用

- 用户要求「复盘」「盘后总结」→ 执行**当日复盘**模式。
- 用户要求「早盘」「快报」「盘前消息」→ 执行**早盘快报**模式。
- 用户给出历史日期，希望复盘指定交易日 → 当日复盘模式（历史日期）。
- 用户直接报出个股名称/代码要求查行情（如「查一下茅台」「光迅科技 工业富联 行情」「帮我看看宁德时代」）→ 执行**快速股价查询**模式。
- 若用户未明确指定模式，默认执行当日复盘。

## 4. 输入约定

### 4.1 通用规则
- 若 `config.yml` 中的 `review.upload.enabled=true`，或用户通过命令行/环境变量显式启用了上传，则执行上报前必须先确认 apiKey 已配置。
- 若已启用上传且本地尚未持久化 `STOCK_REVIEW_API_KEY`，必须先运行 `python ./scripts/stock_review_cli.py set-api-key`。
- 不要把 apiKey 直接写入自然语言回复、日志、markdown、JSON 或命令行历史。
- `apiKey`、`token` 与 `STOCK_REVIEW_API_KEY` 指代同一份接口凭证。

### 4.2 当日复盘
- 默认执行当日复盘；若用户明确指定日期，则执行历史复盘。
- 当日复盘以当日 9:00 至次日 9:00 为一个消息收集周期。
- 历史复盘以用户指定日期的 9:00 至次日 9:00 为消息与盘面收集范围。
- 每日 9:00 至 15:15 交易时段内不执行当日复盘；该限制不适用于历史复盘。
- **跨周末复盘**：若复盘对象日期为周五，且当前时间在周六/周日/周一，则「消息面」应扩展收集周末新闻。将周末重大事件单独列为「周末要闻」板块。

### 4.3 早盘快报
- 收集范围：前一个交易日收盘（15:00）至当前时刻（8:00）的消息。
- 若前一个交易日为周五，收集范围扩展至周五收盘至周一 8:00 的整个周末消息。
- 关注板块/个股必须基于双依据（昨日盘面 + 隔夜消息面），不得仅凭单方面信息判断。

## 5. 输出要求

### 5.1 通用要求（强制）
- **事实回溯**：报告中的每条数据、新闻、个股信息必须能回溯到本次实际采集的原始结果。无法回溯的内容一律删除。
- **JSON 生成方式**：必须通过 Python `json.dump()` 生成，严禁手动拼接 JSON 字符串。JSON 中所有 `changePercent` 字段必须为数字（number）类型——若数据源未提供涨跌幅，填 `0` 并在 `reason` 中标注「数据未获取」。
- **API 上报类型验证**：生成 JSON 后、上报前，**逐字段核对类型**（详见 [`references/review_api.md`](./references/review_api.md) 「常见错误码与排错」章节）。建议直接调 `scripts/validate_review_json.py <file>` 自动校验。
- **输出前事实复查**：逐条核对：
   - 指数涨跌幅与 API 返回值一致？数字类型正确？
   - 板块排名来自实际 API 调用（腾讯 pt）或浏览器 snapshot（JRJ/CLS）？
   - 个股涨跌幅全部来自腾讯 API 实际拉取？没有用板块涨幅推算？
   - 新闻内容可追溯到 JRJ/CLS/Sina 的原始标题或摘要？
- **JSON 字段类型表**：完整字段表见 [`references/review_model.md`](./references/review_model.md)。

### 5.2 当日复盘
- 必须包含：指数表现表格、行业板块涨幅 TOP5、概念板块涨幅 TOP5、领跌板块 TOP5、龙头板块梳理、消息面（分类）。
- 严禁包含 `focusSectors` / `focusStocks`（见 [1.6]）——在 JSON 中发现应立即删除。

### 5.3 早盘快报
- 必须包含：隔夜美股表格、昨日 A 股回顾、盘前重大消息（分类）、今日关注线索、今日关注板块（≤8个）、今日关注个股（≤10只）。
- `todayHot` 必须为对象（即使无盘中数据也要填含空数组的对象）：参考 [`references/review_api.md`](./references/review_api.md) 「`todayHot` 字段必须是对象」章节的最小可用结构。
- `focusSectors` 和 `focusStocks` 必须存在且非空（见 [1.6]）。

## 6. 执行流程

### 6.1 通用步骤
1. 确定运行模式（当日复盘 / 早盘快报）和复盘对象日期。
   - **交易日判断**：先调 `python scripts/is_trading_day.py <date>`，返回 0（交易日）→ 进入步骤 2；返回 1（非交易日）→ 早盘快报仍可生成但需标注「A股休市」，当日复盘应跳过或输出「今日休市，无盘面数据」；返回 2（无法判断）→ 回退到 [`references/holiday_detection.md`](./references/holiday_detection.md) 备用方案。
2. 读取 `config.yml` 确认上传是否启用；若启用，确认 apiKey 已配置。
3. 收集对应模式所需的数据（见下方各模式步骤）。
4. 生成 markdown + JSON。
5. **JSON 类型验证（强制）**：对照 [`references/review_model.md`](./references/review_model.md) 字段表和 [`references/review_api.md`](./references/review_api.md)「常见错误码与排错」章节，逐字段检查类型是否符合 API 期望。
6. 复查事实一致性：每条数据必须能追溯到原始采集结果。
7. 若启用上传，执行 API 上报。

### 6.2 当日复盘

**关键路径**：Sina `/tob/` 综合收评文章 + 东财 `push2` API + 腾讯 API 个股。三者覆盖了 90% 的数据需求，组合效率最高。

1. 收集指数数据（腾讯 API `qt.gtimg.cn` 或可选用 pyTDX 通达信协议）。
2. **【首选·第 1 步】获取 Sina 综合收评文章（`/tob/`）** — 这是**单篇最高效的复盘数据源**：
   - URL：`https://finance.sina.com.cn/tob/YYYY-MM-DD/doc-*.shtml`（从 Sina 财经首页 `/stock/` 搜索 `/tob/YYYY-MM-DD/` 找到）
   - 单篇文章给出：三大指数涨跌幅、涨跌家数（如「下跌个股超4200只」）、热点板块 TOP 排名（按涨幅排序的板块名称 + 部分个股）、跌幅板块、龙头个股涨停清单、消息面分类（政策/产业/海外/公司）、盘后异动、机构观点
   - 适合直接填充 `summary`（取文章首段）、`topSectors.reason`（取文章中各板块描述）、`news[].content`（按分类逐条引用）
   - 收盘后 15:00-16:00 发布，时间窗口匹配复盘流程
   - 2026-06-25 实测验证：单篇文章覆盖了「存储芯片爆发（美光财报催化）、PCB 涨价、券商发力、贵金属跌」等当日全部主线，无需再用多个浏览器页面拼接
3. **【首选·第 2 步】东财 push2 API**（精确板块涨跌幅 + 涨停统计）：
   - 申万行业：`fs=m:90+t:2&fields=f1,f2,f3,f4,f12,f14`，按 `po=1&fid=f3` 取涨幅降序得到 TOP5/TOP10，按 `po=0&fid=f3` 取跌幅 TOP5
   - 概念板块：`fs=m:90+t:3&fields=f1,f2,f3,f4,f12,f14`，同样升降序
   - 个股涨停统计：`fs=m:0+t:6,m:0+t:13,m:0+t:80,m:1+t:2,m:1+t:23&fields=f1,f2,f3,f12,f14`，遍历 200 只统计 `f3 >= 9.9` 和 `f3 <= -9.9` 个数
   - 用于填充 `topSectors[].changePercent` 和 `fallingSectors[].changePercent`（这些字段 Sina 收评文章不直接给出）
4. **【推荐】获取操盘必读文章（/cpbd/）** — 消息面分类最齐全：
   - URL：`https://finance.sina.com.cn/stock/cpbd/YYYY-MM-DD/doc-*.shtml`
   - 包含隔夜美股、宏观政策、行业新闻、公司公告、环球市场等
   - 适合填充 `news[].content`（与 /tob/ 文章互补：/tob/ 偏盘后异动，/cpbd/ 偏全天候消息）
5. 收集个股涨跌幅（腾讯 API 个股代码）— 用于填充 `topSectors[].stocks[]` 和 `fallingSectors[].stocks[]`：
   - 从 Sina 收评文章中提取涨停/跌停龙头股名称后，用腾讯 API 批量查询精确涨跌幅
   - 单次 `urllib.request(qt.gtimg.cn/q=sh603986,sz001309,...)` 可拉 50+ 只股票
   - 严禁用板块涨幅推算个股涨幅（核心约束 1.5.2）
6. 生成完整 markdown + JSON。**输出检查**（具体字段约束见 [5.2] + [1.6]）：
   - ❌ 不得包含「明日关注板块」「明日关注个股」「focusSectors」「focusStocks」章节/字段
   - ✅ 必须包含：指数表格、行业板块 TOP5、概念板块 TOP5、领跌板块 TOP5、龙头梳理、消息面（分类）
   - ✅ 标题格式：`YYYY年M月D日（周X）A股复盘`
   - ✅ JSON 字段类型逐一验证（对照 `references/review_model.md` + `references/review_api.md` 常见错误）
7. 上报 API（先 token 预检 → 再完整 payload POST）。


### 6.3 早盘快报
1. 读取昨日复盘文件：从 `/usr/local/files/docs/stock/YYYY-MM-DD-A股复盘.json`（日期为前一个交易日）中提取昨日盘面数据——指数涨跌幅、领涨领跌板块、涨停/跌停家数、龙头个股走势、资金动向等结构化信息。若昨日复盘文件不存在（如周末/节假日），回退为腾讯 API 或 pyTDX 获取昨日收盘行情。
2. **确定「隔夜美股」应拉哪一天**（先调 `python scripts/resolve_overnight_us.py <trigger_date>`，输出的 `us_data_date` 是真实应使用日期；若 `warning` 不为 null 则按其提示标注美股节假日；`label` 字段直接用于报告标题）。
3. 收集隔夜美股数据（道指/纳指/标普）——按上一步返回的 `us_data_date` 通过腾讯 API `qt.gtimg.cn/q=usDJI,usIXIC,usINX` 拉取对应日数据。数据稳定可靠；亦可尝试 Sina hq API `hq.sinajs.cn/list=gb_dji,gb_ixic,gb_inx` 获取，需添加 Referer 头，注意该 API 可能返回 403 Forbidden——2026-06-26 实测验证；或选用 akshare Sina `index_us_stock_sina()`）。**推荐优先使用腾讯 API**（`qt.gtimg.cn`），其在本服务器上稳定可用。Sina hq API 仅作为备选，若返回 403 则立即回退。对于个股行情（如美光 MU、英伟达 NVDA 等），同样使用腾讯 API `qt.gtimg.cn/q=usMU,usNVDA,...` 批量查询。
4. 收集前日收盘至今的重大消息（**首选**：Sina 操盘必读文章 `finance.sina.com.cn/stock/cpbd/`，详见 [Sina 早报文章参考](./references/sina-morning-articles.md)；**备选**：Sina 财经早报 `finance.sina.com.cn/stock/y/`；**补充**：Sina 7x24 小时快讯 `finance.sina.com.cn/7x24/`（curl+grep 可提取富文本新闻，2026-06-24 验证通过）、JRJ 首页标题 `stock.jrj.com.cn/`、CLS 电报）。
5. 整理今日关注线索（解禁、财报、政策事件等）。
6. 整理今日关注板块：综合昨日复盘中的板块表现（延续强势/回调/轮动）+ 盘前催化方向，筛选 8 个以内，附带名称+短线关注理由。
7. 整理今日关注个股：综合昨日复盘中的个股走势（封板/放量/突破/回调）+ 盘前催化线索，筛选 10 只以内标的，附带代码+名称+短线关注理由。
8. 生成精简 markdown + JSON。**输出检查**（具体字段约束见 [5.3] + [1.6]）：
   - ✅ 必须包含：`focusSectors` 和 `focusStocks` 字段（非空）
   - ✅ `todayHot` 必须是对象（含空数组），不是 null
   - ✅ `markets` 必须是对象（含 `summary`、`indices`、`volume`）
   - ✅ 标题格式：`YYYY年M月D日早盘快报`
   - ❌ 不得包含当日盘中数据
9. 上报 API。（JSON 中 focusSectors 和 focusStocks 按上述第6、7步正常填入）

### 6.4 快速股价查询
1. 解析用户输入：提取个股名称或代码，判断属于哪个市场。
   - A 股：6 位纯数字代码（sh/sz 前缀），如 `sh600519`、`sz002281`
   - 港股：4-5 位数字，如 `0700`、`9988` → 加 `.HK` 后缀
   - 美股：字母代码，如 `AAPL`、`NVDA`
   - 日股：4 位数字，如 `7203` → 加 `.T` 后缀
2. 数据采集：
   - A 股：腾讯 API 批量查询 + 东财公告 API
   - 境外：Yahoo Finance API，URL 格式 `https://query1.finance.yahoo.com/v8/finance/chart/{symbol}?interval=1d&range=2d`
3. **输出字段（强制）**：每条股票必须包含以下三列，缺一不可：
   - **现价**（regularMarketPrice / close）
   - **涨跌幅**（百分比，如 +2.35%、-1.08%）
   - **涨跌额**（绝对值，如 +3.50、-0.80）
   - 外加：最高价、最低价、成交量
4. A 股额外拉取最新 3 条公告（东财公告 API），境外股票可省略公告。
5. 输出格式：`名称(代码) | 现价 | 涨跌幅 | 涨跌额 | 最高/最低 | 成交量 | 备注`
6. 不生成文件，不上报 API，只做内联输出。

## 8. 备注

- 本 skill 适合重复执行的复盘工作流，不适合作为实时交易建议工具。
- 若用户只需要单个字段说明或接口细节，可直接读取相应资源文件，而不必执行完整流程。
- 本 skill 中的 `token` 与 `apiKey` 指代同一份接口凭证。
- 是否执行真实上报由 `config.yml` 中的 `review.upload.enabled` 控制；当其为 `false` 时，允许以“生成 markdown 与 JSON、跳过上报”作为完成态。
- 只有在启用上传时，agent 才必须先满足 apiKey 前置条件，再走脚本上报路径；未启用上传时，不得凭空要求用户提供 apiKey。
- 如果事实依据不足，允许输出"未确认"或直接省略该条，不允许为了完整性伪造事实。
- **Git 提交规则**：本 skill 的修改提交时，「提交」默认指 `commit + push`（推送到 `https://github.com/LittleFogCat/stock-review-skill.git`）。仅当明确说「提交到本地」时才只 commit 不 push。

## 9. 常见陷阱

> 详细原因、代码示例、实测数据已迁移至子文档，SKILL.md 仅保留一句话结论与子文档引用。

### 9.1 工具栈陷阱（patch / 守门员 / 中文引号 / 沙箱 / CLI / token / curl / 浏览器）
Hermes 工具栈使用经验已迁移至部署层文档，不在 Skill 职责范围。

### 9.2 网络数据源阻截（参考 `references/data_source.md`）
腾讯 API `qt.gtimg.cn` 通常稳定；JRJ 首页不可用（见下）；akshare 东财/THS 已被服务端识别；东财 push2 API 自 2026-07-01 完全不可用——回退至 Sina `/cpbd/` + 首页 + snipe + 腾讯 pt 代码组合。

### 9.3 交易日判断（参考 `references/holiday_detection.md`）
非交易日复盘回退到最近交易日；cron 按工作日触发可能误判，使用 Sina 新闻 + Tencent API + pyTDX K线三步检测。

### 9.4 JRJ 不可用
JRJ 首页 `stock.jrj.com.cn/` 仅返回 12-13KB 死页面无数据栏目；子页面频繁 404；CAPTCHA 弹窗可忽略（不阻断内容）。**放弃 JRJ 作为数据入口**，改用 Sina `/cpbd/` + `/tob/` + `/roll/`。

### 9.5 用板块涨幅推算个股涨跌幅导致数据失实（核心约束 1.5.2）
板块涨幅≠个股涨幅，偏离可能很大。个股涨跌幅必须来自腾讯 API 或 pyTDX 的实际拉取结果。

### 9.6 腾讯 API 字段陷阱（参考 `references/tencent-api-field-map.md`）
- 美股 `[3]/[4]` 是当前价/昨收（与 A 股 `[2]/[3]` 互换），涨跌幅在 `[31]/[32]`
- 所有代码族的成交额字段 `[7]` 都返回 `0`，不可用于估算市场总成交额
- pt 板块代码（`pt01801xxx`）覆盖不全，33 板块仅 13 个有数据

### 9.7 早盘周末/跨周五美股含义陷阱（参考 `references/tencent-api-field-map.md` + `references/sina-morning-articles.md`）
周一（08:00）早盘的「隔夜美股」=上周五收盘数据（美股周六周日休市），不是周日晚上→周一凌晨的数据。报告需明确标注「上周五收盘（因周末美股休市）」。

### 9.8 早盘快报生成陷阱
- **领跌板块表格个股重复**：reason 已含个股名时不要拼接个股信息，避免「...工业富联-4.83%（工业富联-4.83%）」
- **浏览器会话冻结**：browser_* 超时切到纯 HTTP/SDK（pytdx / baostock / 腾讯 API）
- **仅凭消息面推测关注标的**：必须先读昨日复盘 JSON 综合「昨日盘面+今早消息」做判断
- **时间错位**：收盘后执行时 API 返回当日数据；优先从昨日复盘 JSON 取前日数据，缺失时回退 Tencent API `prev_close` 字段

### 9.9 美股数据交叉验证（参考 `references/sina-morning-articles.md`）
腾讯 API 美股数据可能返回期货/错误时间戳；新浪操盘必读文章美股数据更可靠（经过编辑团队人工核实）。优先级：操盘必读文章 > 腾讯 API > akshare Sina 美股接口。

### 9.10 Sina 文章采集陷阱（参考 `references/sina-morning-articles.md` + `references/sina-tob-reviews.md`）
- `/cpbd/` `/y/` 在 8:00 前可能 403 → 回退到 `/roll/` 滚动新闻或首页标题提取
- `/tob/` 文章正文通过 JS 动态加载，`<p>` 标签仅返回专题语 → 优先 `<meta name="description">` + `/cpbd/` + `/bxjj/` 组合
- 收评文章分布在 `/tob/` `/bxjj/` `/snipe/` 三条路径，需全部检查；bxjj 标题须含「ETF收评」或「收评」字样

### 9.11 数据源当前状态（参考 `references/data_source.md` + `references/cron_setup.md`）
- 东财 push2 API：2026-07-01 起不可用
- JRJ 首页：死页面，已放弃
- Cron 空闲超时：600s 模型 API hang → Gateway 看门狗
- Token 预检：上报前 ping payload 避免浪费 LLM 调用
- `no_agent=true` 脚本：stdout 即投递，异常才输出

### 9.12 Cron 部署层细节（参考 `references/cron_setup.md`）
final response 推送语义、互斥锁、read_file 行号陷阱、明日观察点改写、节假日时间框架、用户指令冲突处理 —— 全部在 cron_setup.md 对应章节。skill 本身保留 1.6 输出章节约束第 1 条（当日复盘不包含 focusSectors/focusStocks）作为核心约束。

### 9.13 上报方式陷阱
Python 上报（`write_file + python3 /tmp/upload.py`）无 shell 变量替换问题，比 curl 更可靠（覆盖 1.8 上报约束第 3 条）；cron 模式推荐 Python。
